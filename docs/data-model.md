# The Product Profile Data Model

## Overview

A ProductProfile is a structured representation of how a product converts users into revenue. It is the core data type in Basesignal, generated by scanning a product's public-facing website and analyzing the crawled content with an LLM.

Every profile captures what the product is, who it serves, how users progress through it, and how it generates value. The profile is designed to be both human-readable and machine-processable -- you can export it as markdown for a quick overview or consume the JSON programmatically for analysis and tooling.

## Profile Structure

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `basesignal_version` | `string` | Yes | Schema version (e.g., `"1.0"`) |
| `metadata` | `ProfileMetadata` | No | Creation timestamp, update timestamp, source URL |
| `identity` | `CoreIdentity` | No | Product name, description, target customer, business model |
| `revenue` | `RevenueArchitecture` | No | Pricing model, tiers, expansion paths, contraction risks |
| `entities` | `EntityModel` | No | Domain objects and their relationships |
| `journey` | `UserJourney` | No | User lifecycle stages from discovery to expansion |
| `definitions` | `DefinitionsMap` | No | Activation, first value, active, at-risk, and churn criteria |
| `outcomes` | `OutcomesSection` | No | Business outcomes linked to product features |
| `metrics` | `MetricsSection` | No | Key performance metrics organized by P&L layer |
| `outputs` | `Outputs` | No | ICP profiles, activation map, measurement spec |
| `completeness` | `number` | Yes | Fraction of sections populated (0-1) |
| `overallConfidence` | `number` | Yes | Weighted average confidence across all sections (0-1) |

All optional sections become populated as the scan progresses. A scan that only finds a homepage and pricing page may populate `identity` and `revenue` but leave `definitions` empty. The `completeness` field tracks how much of the profile has been filled.

## Sections

### Core Identity

Captures the foundational product identity: what the product is, who it serves, and how it makes money. Populated from homepage, about, and features pages.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `productName` | `string` | Yes | The product's name |
| `description` | `string` | Yes | Brief product description |
| `targetCustomer` | `string` | Yes | Who the product serves |
| `businessModel` | `string` | Yes | How the product generates revenue |
| `industry` | `string` | No | Industry vertical |
| `companyStage` | `string` | No | Company maturity stage |
| `confidence` | `number` | Yes | Extraction confidence (0-1) |
| `evidence` | `Evidence[]` | Yes | Source URLs and excerpts |

Example from the [Linear profile fixture](./fixtures/linear-profile.json):

```json
{
  "productName": "Linear",
  "description": "Modern project management tool built for software teams",
  "targetCustomer": "Software development teams at startups and mid-market companies",
  "businessModel": "Per-seat SaaS subscription",
  "industry": "Developer Tools",
  "companyStage": "Growth",
  "confidence": 0.92,
  "evidence": [
    { "url": "https://linear.app/", "excerpt": "Linear is a better way to build software" },
    { "url": "https://linear.app/pricing", "excerpt": "Per member per month" }
  ]
}
```

### Revenue Architecture

Describes how the product captures value: pricing model, billing mechanics, tier structure, upsell paths, and downgrade risks.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `model` | `string` | Yes | Revenue model (e.g., "Per-seat SaaS subscription") |
| `billingUnit` | `string` | No | What the customer pays per (e.g., "seat", "usage") |
| `hasFreeTier` | `boolean` | Yes | Whether a free plan exists |
| `tiers` | `PricingTier[]` | Yes | Available pricing tiers |
| `expansionPaths` | `string[]` | Yes | Upsell/cross-sell vectors |
| `contractionRisks` | `string[]` | Yes | Downgrade trigger scenarios |
| `confidence` | `number` | Yes | Extraction confidence (0-1) |
| `evidence` | `Evidence[]` | Yes | Source URLs and excerpts |

Each `PricingTier` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | `string` | Yes | Tier name (e.g., "Free", "Standard") |
| `price` | `string` | Yes | Price description (e.g., "$8/user/month") |
| `features` | `string[]` | Yes | Features included in this tier |

Example:

```json
{
  "model": "Per-seat SaaS subscription",
  "billingUnit": "seat",
  "hasFreeTier": true,
  "tiers": [
    { "name": "Free", "price": "$0/month", "features": ["Up to 250 issues", "Basic integrations", "All core features"] },
    { "name": "Standard", "price": "$8/user/month", "features": ["Unlimited issues", "Cycles", "Priority support", "Custom workflows"] },
    { "name": "Plus", "price": "$14/user/month", "features": ["Advanced analytics", "SLA guarantees", "SAML SSO", "Audit log"] }
  ],
  "expansionPaths": [
    "Free to Standard: teams hitting the 250 issue limit",
    "Standard to Plus: enterprises needing SSO and compliance",
    "Seat expansion: team growth adds per-user revenue"
  ],
  "contractionRisks": [
    "Team downsizing reduces seat count",
    "Switching to competing tool (Jira, Shortcut)",
    "Plus to Standard downgrade if advanced features unused"
  ],
  "confidence": 0.91,
  "evidence": [
    { "url": "https://linear.app/pricing", "excerpt": "Free for small teams. Standard at $8/user/month." }
  ]
}
```

### User Journey

Maps the progression a user follows from first touch through core usage and expansion. Stages are ordered and describe the user's experience at each step.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `stages` | `JourneyStage[]` | Yes | Ordered lifecycle stages |
| `confidence` | `number` | Yes | Extraction confidence (0-1) |
| `evidence` | `Evidence[]` | Yes | Source URLs and excerpts |

Each `JourneyStage` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | `string` | Yes | Stage name (e.g., "Signup", "Activation") |
| `description` | `string` | Yes | What happens in this stage |
| `order` | `number` | Yes | Position in the journey (0-indexed) |

Example:

```json
{
  "stages": [
    { "name": "Discovery", "description": "User visits website and explores features", "order": 0 },
    { "name": "Signup", "description": "Creates account and sets up workspace", "order": 1 },
    { "name": "Onboarding", "description": "Creates first project, invites team members", "order": 2 },
    { "name": "Activation", "description": "Team creates and triages first issues", "order": 3 },
    { "name": "Core Usage", "description": "Daily sprint management and issue tracking", "order": 4 },
    { "name": "Expansion", "description": "Adopts roadmaps, cycles, and integrations", "order": 5 }
  ],
  "confidence": 0.85,
  "evidence": [
    { "url": "https://linear.app/docs/getting-started", "excerpt": "Create your workspace and invite your team" }
  ]
}
```

### Definitions

Behavioral definitions for user lifecycle states. These translate qualitative product understanding into measurable criteria that drive the P&L framework.

The `DefinitionsMap` contains five optional definitions:

| Field | Type | Description |
|-------|------|-------------|
| `activation` | `ActivationDefinition` | When a user is considered activated (supports legacy flat and multi-level formats) |
| `firstValue` | `LifecycleDefinition` | The "aha moment" when the user first sees the product's core benefit |
| `active` | `LifecycleDefinition` | Ongoing engagement criteria for a healthy user |
| `atRisk` | `LifecycleDefinition` | Disengagement signals predicting potential churn |
| `churn` | `LifecycleDefinition` | Loss criteria -- when a user is considered gone |

#### Activation (Multi-Level Format)

The multi-level format models activation as a progressive, multi-step process. Each level has a signal strength indicating how strong the activation signal is.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `levels` | `ActivationLevelDef[]` | Yes | Ordered activation levels |
| `primaryActivation` | `number` | No | Which level is the primary activation threshold |
| `overallConfidence` | `number` | Yes | Overall confidence across levels (0-1) |

Each `ActivationLevelDef` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `level` | `number` | Yes | Level number (1, 2, 3, ...) |
| `name` | `string` | Yes | Human-readable level name |
| `signalStrength` | `SignalStrength` | Yes | `"weak"`, `"medium"`, `"strong"`, or `"very_strong"` |
| `criteria` | `ActivationCriterion[]` | Yes | Structured criteria for this level |
| `reasoning` | `string` | Yes | Why this level matters |
| `confidence` | `number` | Yes | Confidence in this level (0-1) |
| `evidence` | `Evidence[]` | Yes | Source URLs and excerpts |

Each `ActivationCriterion` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `action` | `string` | Yes | The user action to measure (e.g., `"create_issue"`) |
| `count` | `number` | Yes | How many times the action must occur |
| `timeWindow` | `string` | No | Time window for the action (e.g., `"7d"`) |

Example (multi-level activation from the Linear fixture):

```json
{
  "levels": [
    {
      "level": 1,
      "name": "Setup Complete",
      "signalStrength": "weak",
      "criteria": [
        { "action": "create_workspace", "count": 1 },
        { "action": "invite_member", "count": 1 }
      ],
      "reasoning": "Workspace setup with at least one invited member shows intent to use the product collaboratively",
      "confidence": 0.8,
      "evidence": [
        { "url": "https://linear.app/docs/getting-started", "excerpt": "Start by creating your workspace" }
      ]
    },
    {
      "level": 2,
      "name": "First Workflow",
      "signalStrength": "medium",
      "criteria": [
        { "action": "create_issue", "count": 5, "timeWindow": "7d" },
        { "action": "move_issue_status", "count": 3, "timeWindow": "7d" }
      ],
      "reasoning": "Creating and triaging issues shows the team is adopting Linear for real work",
      "confidence": 0.85,
      "evidence": [
        { "url": "https://linear.app/docs/issues", "excerpt": "Issues are the building blocks of Linear" }
      ]
    }
  ],
  "primaryActivation": 2,
  "overallConfidence": 0.85
}
```

#### Activation (Legacy Flat Format)

The legacy format uses a flat list of string criteria. Both formats are valid -- consumers should check for the presence of a `levels` field to distinguish them.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `criteria` | `string[]` | Yes | Human-readable activation criteria |
| `timeWindow` | `string` | No | Time window for all criteria (e.g., `"14d"`) |
| `reasoning` | `string` | Yes | Why these criteria indicate activation |
| `confidence` | `number` | Yes | Confidence in the definition (0-1) |
| `source` | `string` | Yes | How the definition was derived |
| `evidence` | `Evidence[]` | Yes | Source URLs and excerpts |

| Field | Type | Required | Description |
|-------|------|----------|-------------|

### Entity Model

Describes the core data objects a product manages and the relationships between them -- the product's domain model as understood from public-facing pages.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `items` | `EntityItem[]` | Yes | Entities discovered in the product |
| `relationships` | `EntityRelationship[]` | Yes | Relationships between entities |
| `confidence` | `number` | Yes | Extraction confidence (0-1) |
| `evidence` | `Evidence[]` | Yes | Source URLs and excerpts |

Each `EntityItem` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | `string` | Yes | Entity name (e.g., "Issue", "Project") |
| `type` | `string` | Yes | Classification (e.g., "core", "container", "actor") |
| `properties` | `string[]` | Yes | Key properties of this entity |

Each `EntityRelationship` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `from` | `string` | Yes | Source entity name |
| `to` | `string` | Yes | Target entity name |
| `type` | `string` | Yes | Relationship type (e.g., "has many", "contains") |

Example:

```json
{
  "items": [
    { "name": "Issue", "type": "core", "properties": ["title", "status", "priority", "assignee", "labels", "estimate"] },
    { "name": "Project", "type": "container", "properties": ["name", "lead", "status", "targetDate", "progress"] },
    { "name": "Cycle", "type": "time-bound", "properties": ["name", "startDate", "endDate", "issues", "completedIssues"] }
  ],
  "relationships": [
    { "from": "Workspace", "to": "Project", "type": "has many" },
    { "from": "Project", "to": "Issue", "type": "has many" },
    { "from": "Cycle", "to": "Issue", "type": "contains" }
  ],
  "confidence": 0.88,
  "evidence": [
    { "url": "https://linear.app/docs/issues", "excerpt": "Issues are the building blocks of Linear" }
  ]
}
```

### Metric Catalog

Key performance metrics organized by the four layers of the Basesignal P&L framework.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `items` | `MetricItem[]` | Yes | Catalog of metrics |
| `confidence` | `number` | Yes | Extraction confidence (0-1) |
| `evidence` | `Evidence[]` | Yes | Source URLs and excerpts |

Each `MetricItem` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | `string` | Yes | Metric name (e.g., "Weekly Active Users") |
| `category` | `string` | Yes | P&L layer: `"reach"`, `"engagement"`, `"value_delivery"`, or `"value_capture"` |
| `formula` | `string` | No | How the metric is calculated |
| `linkedTo` | `string[]` | Yes | Features this metric relates to |

P&L framework mapping:

| Category | What It Measures | Example Metrics |
|----------|-----------------|-----------------|
| `reach` | New user volume, trial starts | Signup Rate, Website Visitors |
| `engagement` | Active rate, feature adoption, usage intensity | Weekly Active Users, Feature Adoption Rate |
| `value_delivery` | Activation, first value delivery, account health | Activation Rate, Active Account Rate |
| `value_capture` | Conversion, retention, expansion | Trial-to-Paid Conversion, Net Revenue Retention |

Example:

```json
{
  "items": [
    { "name": "Weekly Active Users", "category": "engagement", "formula": "COUNT(DISTINCT users WHERE active_last_7d)", "linkedTo": ["Issue tracking", "Sprint planning"] },
    { "name": "Trial-to-Paid Conversion", "category": "value_capture", "formula": "paid_workspaces / trial_workspaces", "linkedTo": ["Onboarding flow", "Pricing page"] },
    { "name": "Signup Rate", "category": "reach", "linkedTo": ["Landing page", "Pricing page"] }
  ],
  "confidence": 0.78,
  "evidence": [
    { "url": "https://linear.app/pricing", "excerpt": "Start free, upgrade when ready" }
  ]
}
```

### Outputs

Three output artifacts generated from the analysis pipeline. These are higher-order analyses built on top of the core profile sections.

#### ICP Profiles

Ideal Customer Profile segments derived from value moment analysis. Each ICP describes a specific user persona, their priorities, activation triggers, pain points, and success metrics.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | `string` | Yes | Unique identifier |
| `name` | `string` | Yes | Persona name (e.g., "Engineering Team Lead") |
| `description` | `string` | Yes | Who this persona is and what they do |
| `value_moment_priorities` | `ValueMomentPriority[]` | Yes | Ranked value moments for this persona |
| `activation_triggers` | `string[]` | Yes | What drives this persona to try the product |
| `pain_points` | `string[]` | Yes | Problems this persona faces |
| `success_metrics` | `string[]` | Yes | How this persona measures success |
| `confidence` | `number` | Yes | Confidence in this ICP (0-1) |
| `sources` | `string[]` | Yes | Source URLs |

Example:

```json
{
  "id": "icp-eng-lead",
  "name": "Engineering Team Lead",
  "description": "Leads a team of 5-15 engineers at a startup or scale-up, responsible for shipping velocity and sprint planning",
  "value_moment_priorities": [
    { "moment_id": "vm-sprint-velocity", "priority": 1, "relevance_reason": "Directly responsible for team output and cadence" }
  ],
  "activation_triggers": [
    "Migrating from Jira or another tool that feels slow",
    "Scaling team and needing better sprint management"

  "pain_points": [
    "Jira is too slow and cluttered for fast-moving teams",

  ],
  "success_metrics": [
    "Team creates and closes 50+ issues per cycle",

  "sources": ["https://linear.app/", "https://linear.app/customers"]
}
```

#### Activation Map

A multi-stage model of how users progress from first touch to full activation, with transitions and drop-off risks at each stage.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `stages` | `ActivationStage[]` | Yes | Ordered activation stages |
| `transitions` | `StageTransition[]` | Yes | How users move between stages |
| `primary_activation_level` | `number` | Yes | The level considered "activated" |
| `confidence` | `number` | Yes | Confidence in the map (0-1) |
| `sources` | `string[]` | Yes | Source URLs |

Each `ActivationStage` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `level` | `number` | Yes | Stage level number |
| `name` | `string` | Yes | Stage name |
| `signal_strength` | `SignalStrength` | Yes | `"weak"`, `"medium"`, `"strong"`, or `"very_strong"` |
| `trigger_events` | `string[]` | Yes | Events that indicate entering this stage |
| `value_moments_unlocked` | `string[]` | Yes | Value moments available at this stage |
| `drop_off_risk` | `string` | Yes | `"low"`, `"medium"`, or `"high"` |
| `drop_off_reasons` | `string[]` | No | Why users might drop off at this stage |

Each `StageTransition` has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `from_level` | `number` | Yes | Source stage level |
| `to_level` | `number` | Yes | Target stage level |
| `trigger_events` | `string[]` | Yes | Events that trigger the transition |
| `typical_timeframe` | `string` | No | How long the transition typically takes |

| `userStateModel` | `UserState[]` | Yes | User lifecycle state definitions |
| `confidence` | `number` | Yes | Confidence in the spec (0-1) |

Each `TrackingEvent` has:

| Field | Type | Required | Description |

## Confidence and Evidence

Every section in a ProductProfile includes a `confidence` score (0-1) and an `evidence` array.

**Confidence** reflects how reliably the data was extracted from the crawled pages:

| Range | Meaning |
|-------|---------|
| 0.8 - 1.0 | High: multiple corroborating sources, explicit data found on pages |
| 0.5 - 0.79 | Medium: fewer sources, some inference from indirect signals |
| 0.0 - 0.49 | Low: single source, significant inference required |

**Evidence** links each extracted data point back to a specific URL and text excerpt from the crawled page:

```json
{
  "url": "https://linear.app/pricing",
  "excerpt": "Free for small teams. Standard at $8/user/month."
}
```

Evidence serves two purposes: it provides an audit trail for the extraction, and it lets users verify or correct the analysis by visiting the source pages.

## Schema Versioning

Every profile carries a `basesignal_version` field (e.g., `"1.0"`) that identifies which version of the schema was used to produce it.

- **Minor bumps** (1.0 to 1.1): Additive changes only -- new optional fields, new enum values. Old profiles remain valid.
- **Major bumps** (1.0 to 2.0): Breaking changes -- removed fields, type changes, renames. Old profiles may not validate.

The `checkVersion()` utility from `@basesignal/core` compares a profile's version against the library version:

```typescript
import { checkVersion } from "@basesignal/core";

const status = checkVersion(profile.basesignal_version);
// "compatible" | "needs_migration" | "incompatible"
```

Unknown fields are preserved, not rejected. A 1.0 reader parsing a 1.1 profile passes through fields it does not recognize.
