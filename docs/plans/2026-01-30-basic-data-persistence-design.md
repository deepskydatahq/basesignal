# Basic Data Persistence Design

## Overview

Design the Convex schema for the MCP product knowledge data model. A single `productProfiles` document per product stores all analyzed sections (identity, revenue, entities, journey, definitions, outcomes, metrics) with confidence scores and evidence links. A separate `scanJobs` table tracks crawl/analysis state.

## Problem Statement

The MCP server needs to persist structured product knowledge that's generated by the scan pipeline and refined through conversation. The schema must support: incremental building (sections fill in as analysis progresses), confidence tracking (AI-inferred vs. user-validated), and efficient retrieval (get full profile or specific sections).

## Proposed Solution

### Architecture Decision

- **Database**: Convex (same deployment as existing web app)
- **Schema approach**: Single profile document per product (not one table per section)
- **Confidence tracking**: Per-section for most data, per-field for definitions
- **Relationship to products table**: `productProfiles` references `products` via `productId`

### New Tables

Two new tables added to `convex/schema.ts`:

#### 1. `productProfiles` — The Knowledge Store

One document per product. All sections are optional so the profile builds up incrementally as the scan fills in each section.

```typescript
productProfiles: defineTable({
  productId: v.id("products"),

  // === Core Identity ===
  identity: v.optional(v.object({
    productName: v.string(),
    description: v.string(),
    targetCustomer: v.string(),
    businessModel: v.string(),
    industry: v.optional(v.string()),
    companyStage: v.optional(v.string()),
    confidence: v.number(),
    evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
  })),

  // === Revenue Architecture ===
  revenue: v.optional(v.object({
    model: v.string(),
    billingUnit: v.optional(v.string()),
    hasFreeTier: v.boolean(),
    tiers: v.array(v.object({
      name: v.string(),
      price: v.string(),
      features: v.array(v.string()),
    })),
    expansionPaths: v.array(v.string()),
    contractionRisks: v.array(v.string()),
    confidence: v.number(),
    evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
  })),

  // === Entity Model ===
  entities: v.optional(v.object({
    items: v.array(v.object({
      name: v.string(),
      type: v.string(),
      properties: v.array(v.string()),
    })),
    relationships: v.array(v.object({
      from: v.string(),
      to: v.string(),
      type: v.string(),
    })),
    confidence: v.number(),
    evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
  })),

  // === Journey Stages ===
  journey: v.optional(v.object({
    stages: v.array(v.object({
      name: v.string(),
      description: v.string(),
      order: v.number(),
    })),
    confidence: v.number(),
    evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
  })),

  // === Definitions (per-field confidence + source tracking) ===
  definitions: v.optional(v.object({
    activation: v.optional(v.object({
      criteria: v.array(v.string()),
      timeWindow: v.optional(v.string()),
      reasoning: v.string(),
      confidence: v.number(),
      source: v.string(),
      evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
    })),
    firstValue: v.optional(v.object({
      description: v.string(),
      criteria: v.array(v.string()),
      reasoning: v.string(),
      confidence: v.number(),
      source: v.string(),
      evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
    })),
    active: v.optional(v.object({
      criteria: v.array(v.string()),
      timeWindow: v.optional(v.string()),
      reasoning: v.string(),
      confidence: v.number(),
      source: v.string(),
      evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
    })),
    atRisk: v.optional(v.object({
      criteria: v.array(v.string()),
      timeWindow: v.optional(v.string()),
      reasoning: v.string(),
      confidence: v.number(),
      source: v.string(),
      evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
    })),
    churn: v.optional(v.object({
      criteria: v.array(v.string()),
      timeWindow: v.optional(v.string()),
      reasoning: v.string(),
      confidence: v.number(),
      source: v.string(),
      evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
    })),
  })),

  // === Outcomes ===
  outcomes: v.optional(v.object({
    items: v.array(v.object({
      description: v.string(),
      type: v.string(),
      linkedFeatures: v.array(v.string()),
    })),
    confidence: v.number(),
    evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
  })),

  // === Metrics ===
  metrics: v.optional(v.object({
    items: v.array(v.object({
      name: v.string(),
      category: v.string(),
      formula: v.optional(v.string()),
      linkedTo: v.array(v.string()),
    })),
    confidence: v.number(),
    evidence: v.array(v.object({ url: v.string(), excerpt: v.string() })),
  })),

  // === Computed Meta ===
  completeness: v.number(),
  overallConfidence: v.number(),
  createdAt: v.number(),
  updatedAt: v.number(),
})
  .index("by_product", ["productId"])
```

#### 2. `scanJobs` — Crawl & Analysis State

Tracks the async scan pipeline. One job per scan request.

```typescript
scanJobs: defineTable({
  productId: v.id("products"),
  userId: v.id("users"),
  status: v.string(),            // "crawling" | "extracting" | "analyzing" | "complete" | "failed"
  url: v.string(),               // The URL being scanned

  // Progress tracking
  pagesCrawled: v.number(),
  pagesTotal: v.optional(v.number()),
  currentPhase: v.string(),

  // Results
  crawledPages: v.optional(v.array(v.object({
    url: v.string(),
    pageType: v.optional(v.string()),
    title: v.optional(v.string()),
  }))),

  // Discovered resources
  discoveredDocs: v.optional(v.string()),    // docs site URL if found
  discoveredPricing: v.optional(v.string()), // pricing page URL if found

  // Error tracking
  error: v.optional(v.string()),

  // Timing
  startedAt: v.number(),
  completedAt: v.optional(v.number()),
})
  .index("by_product", ["productId"])
  .index("by_user", ["userId"])
```

### Convex Functions

```
convex/productProfiles.ts
├── create: mutation               // Create empty profile for product
├── get: query                     // Get full profile by productId
├── getSection: query              // Get specific section by name
├── updateSection: mutation        // Update a single section (merge)
├── updateDefinition: mutation     // Update a specific definition
├── validateSection: mutation      // Mark section as user-validated (confidence → 1.0)
├── calculateCompleteness: mutation // Recalculate completeness/confidence scores
└── remove: mutation               // Delete profile (cascade from product delete)

convex/scanJobs.ts
├── create: mutation               // Start a new scan job
├── get: query                     // Get job status
├── updateProgress: mutation       // Update crawl progress
├── complete: mutation             // Mark job complete
├── fail: mutation                 // Mark job failed with error
└── listByProduct: query           // List scan history for a product
```

### Key Design Decisions

**Single document vs. normalized tables:**
A single profile document is simpler to read/write and maps directly to the MCP `get_profile()` tool. The tradeoff is that very large profiles could hit Convex document size limits (1MB), but a product profile with all sections filled in is well under that.

**Optional sections:**
Every section is `v.optional()`. The profile starts empty and fills in as the scan pipeline processes each analysis step. This supports incremental building and partial profiles.

**Confidence + source tracking:**
- Section-level: `confidence` (0-1) and `evidence` array on each section
- Definition-level: Additional `source` field ("ai-inferred" | "user-validated") on each definition
- When a user validates/updates a section, `confidence` goes to 1.0 and `source` becomes "user-validated"

**Evidence as inline arrays:**
Evidence (URL + excerpt) is stored inline within each section rather than in a separate table. This keeps reads simple and avoids joins. The tradeoff is some duplication if the same page informs multiple sections — acceptable.

**Scan jobs as separate table:**
Scan state is transient (progress updates during crawl) while profile state is persistent (the analyzed result). Keeping them separate means we can query scan history without polluting the profile document.

### Completeness Calculation

```
completeness = (filled sections / total sections)

Sections: identity, revenue, entities, journey,
          definitions.activation, definitions.firstValue,
          definitions.active, definitions.churn,
          outcomes, metrics

= count of non-null sections / 10
```

### Overall Confidence Calculation

```
overallConfidence = average(confidence of all filled sections)
```

Weighted average could be considered later (definitions matter more than identity).

## Alternatives Considered

### Table per section
Each section in its own Convex table: `productIdentities`, `productRevenue`, `productJourneys`, etc. More normalized but requires multiple queries to assemble a full profile and more mutations to update. Adds complexity without clear benefit at this scale.

### JSON blob
Store the entire profile as a single JSON string field. Maximum flexibility but loses Convex's type validation, makes partial queries impossible, and no schema enforcement.

### PostgreSQL
Use a relational database with proper normalization. Would work well but introduces a new infrastructure dependency. The decision to keep Convex was made in Story 1.3 brainstorming.

## Open Questions

1. **Document size limits**: Convex has a 1MB document limit. Need to verify a fully populated profile stays well under this. Estimate: ~50KB for a comprehensive profile — plenty of room.
2. **Crawled content storage**: Raw HTML from the crawl may be too large for Convex documents. May need Convex file storage or external object storage for raw crawl data. The `scanJobs` table only stores metadata, not raw content.
3. **Version history**: The profile document gets overwritten on updates. History/versioning (Story 7.2) will need a separate approach — either a changelog table or snapshots. Not needed for MVP.

## Success Criteria

- Schema deployed to Convex without errors
- Profile can be created, read, and updated section by section
- Scan jobs can be created and progress tracked
- Completeness and confidence scores calculate correctly
- Profile survives server restarts (Convex persistence)
- All operations enforce ownership (only the product owner can access)
