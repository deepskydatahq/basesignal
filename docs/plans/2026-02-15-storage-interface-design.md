# Storage Interface Definition and SQLite Adapter Design

## Overview

Define a `StorageAdapter` interface with five operations (save, load, list, delete, search) and implement an SQLite adapter using `better-sqlite3` as the zero-config default. The interface stores `ProductProfile` documents as JSON blobs with indexed metadata columns for search, mapping directly from the existing Convex `productProfiles` table shape.

## Problem Statement

Basesignal currently stores all data in Convex, a hosted serverless backend. The open source distribution (M008) needs self-hostable storage that works without external services. Users running `basesignal scan <url>` from the CLI need profiles to persist somewhere locally. The storage layer must be swappable so the community can add Postgres, Turso, or other adapters without touching core code.

## Expert Perspectives

### Technical Architect

The core question is: what is the minimal interface that covers all read/write patterns the CLI and MCP server need? The answer is five operations: save, load, list, delete, and search. This maps 1:1 to how the existing Convex layer is used -- `productProfiles.create`/`updateSection` collapse into `save`, `get` becomes `load`, and the products list page becomes `list`. The JSON-blob-with-metadata-columns pattern is the right call -- it avoids an ORM, avoids schema migrations for profile shape changes, and keeps the SQLite schema at exactly one table. The profile type system lives in `@basesignal/core` (M008-E001-S002); storage just persists and retrieves those documents.

### Simplification Reviewer

**Verdict: APPROVED** -- with one cut.

What to keep:
- Single interface, five methods. This is the minimum.
- JSON blob storage with metadata columns. Avoids premature normalization.
- `better-sqlite3` synchronous API. No async ceremony for a local database.
- `:memory:` for tests. No file I/O in test suite.

What to cut:
- **No migration framework.** The schema is one table. If the table shape changes, drop and recreate -- profiles are derived data that can be regenerated from a scan. A migration system is premature complexity for v1.

What to watch:
- The `search()` method. Keep it dead simple (LIKE on name and url). Do not add full-text search, faceted search, or scoring. If someone needs that, they'll use Postgres.
- No connection pooling. `better-sqlite3` is synchronous and single-connection. That's a feature, not a limitation, for a CLI tool.

## Proposed Solution

### Storage Interface

```typescript
// packages/storage/src/types.ts

/**
 * Minimal metadata returned by list() and search().
 * Avoids loading full profile documents for listing views.
 */
export interface ProfileSummary {
  id: string;
  name: string;
  url: string;
  completeness: number;
  updatedAt: number;
}

/**
 * Storage adapter interface. All implementations must satisfy this contract.
 * Profiles are ProductProfile documents from @basesignal/core.
 */
export interface StorageAdapter {
  /** Persist a profile. Returns the profile ID (generated if not present). */
  save(profile: ProductProfile): Promise<string>;

  /** Load a profile by ID. Returns null if not found. */
  load(id: string): Promise<ProductProfile | null>;

  /** List all profiles with summary metadata. */
  list(): Promise<ProfileSummary[]>;

  /** Delete a profile by ID. Returns true if it existed. */
  delete(id: string): Promise<boolean>;

  /** Search profiles by name or URL substring. */
  search(query: string): Promise<ProfileSummary[]>;

  /** Clean up resources (close database connections, etc.). */
  close(): void;
}
```

Note: Methods return `Promise` even though `better-sqlite3` is synchronous. This keeps the interface compatible with async adapters (Postgres, HTTP-backed, etc.). The SQLite adapter wraps synchronous calls in `Promise.resolve()`.

### SQLite Schema

One table, one index:

```sql
CREATE TABLE IF NOT EXISTS profiles (
  id          TEXT PRIMARY KEY,
  name        TEXT NOT NULL,
  url         TEXT NOT NULL,
  completeness REAL NOT NULL DEFAULT 0,
  data        TEXT NOT NULL,           -- JSON-serialized ProductProfile
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_profiles_name ON profiles(name);
CREATE INDEX IF NOT EXISTS idx_profiles_url ON profiles(url);
```

Design rationale:
- `id` is a UUID generated by the adapter on first save (not a Convex ID).
- `name` and `url` are denormalized from the JSON for indexed search.
- `data` stores the full `ProductProfile` as a JSON string. This means profile schema changes (new sections, renamed fields) require zero SQLite migrations.
- `completeness` is denormalized for list/sort without JSON parsing.
- Timestamps are Unix milliseconds (matching Convex convention).

### SQLite Adapter Implementation

```typescript
// packages/storage/src/sqlite.ts

import Database from "better-sqlite3";
import { randomUUID } from "node:crypto";
import { mkdirSync } from "node:fs";
import { dirname } from "node:path";
import { homedir } from "node:os";
import { join } from "node:path";

const DEFAULT_DB_PATH = join(homedir(), ".basesignal", "data.db");

export interface SQLiteStorageOptions {
  /** Path to the SQLite database file. Use ':memory:' for in-memory. */
  path?: string;
}

export class SQLiteStorage implements StorageAdapter {
  private db: Database.Database;

  constructor(options: SQLiteStorageOptions = {}) {
    const dbPath = options.path ?? DEFAULT_DB_PATH;

    // Auto-create directory for file-based databases
    if (dbPath !== ":memory:") {
      mkdirSync(dirname(dbPath), { recursive: true });
    }

    this.db = new Database(dbPath);
    this.db.pragma("journal_mode = WAL");
    this.db.pragma("foreign_keys = ON");
    this.initialize();
  }

  private initialize(): void {
    this.db.exec(`
      CREATE TABLE IF NOT EXISTS profiles (
        id          TEXT PRIMARY KEY,
        name        TEXT NOT NULL,
        url         TEXT NOT NULL,
        completeness REAL NOT NULL DEFAULT 0,
        data        TEXT NOT NULL,
        created_at  INTEGER NOT NULL,
        updated_at  INTEGER NOT NULL
      );
      CREATE INDEX IF NOT EXISTS idx_profiles_name ON profiles(name);
      CREATE INDEX IF NOT EXISTS idx_profiles_url ON profiles(url);
    `);
  }

  async save(profile: ProductProfile): Promise<string> {
    const now = Date.now();
    const id = profile.id ?? randomUUID();
    const name = profile.identity?.productName ?? "";
    const url = profile.metadata?.url ?? "";
    const completeness = profile.completeness ?? 0;

    const profileWithId = { ...profile, id };
    const data = JSON.stringify(profileWithId);

    const stmt = this.db.prepare(`
      INSERT INTO profiles (id, name, url, completeness, data, created_at, updated_at)
      VALUES (?, ?, ?, ?, ?, ?, ?)
      ON CONFLICT(id) DO UPDATE SET
        name = excluded.name,
        url = excluded.url,
        completeness = excluded.completeness,
        data = excluded.data,
        updated_at = excluded.updated_at
    `);

    stmt.run(id, name, url, completeness, data, now, now);
    return id;
  }

  async load(id: string): Promise<ProductProfile | null> {
    const row = this.db.prepare("SELECT data FROM profiles WHERE id = ?").get(id) as
      | { data: string }
      | undefined;
    if (!row) return null;
    return JSON.parse(row.data);
  }

  async list(): Promise<ProfileSummary[]> {
    const rows = this.db
      .prepare("SELECT id, name, url, completeness, updated_at FROM profiles ORDER BY updated_at DESC")
      .all() as Array<{ id: string; name: string; url: string; completeness: number; updated_at: number }>;

    return rows.map((r) => ({
      id: r.id,
      name: r.name,
      url: r.url,
      completeness: r.completeness,
      updatedAt: r.updated_at,
    }));
  }

  async delete(id: string): Promise<boolean> {
    const result = this.db.prepare("DELETE FROM profiles WHERE id = ?").run(id);
    return result.changes > 0;
  }

  async search(query: string): Promise<ProfileSummary[]> {
    const pattern = `%${query}%`;
    const rows = this.db
      .prepare(
        "SELECT id, name, url, completeness, updated_at FROM profiles WHERE name LIKE ? OR url LIKE ? ORDER BY updated_at DESC"
      )
      .all(pattern, pattern) as Array<{
        id: string; name: string; url: string; completeness: number; updated_at: number;
      }>;

    return rows.map((r) => ({
      id: r.id,
      name: r.name,
      url: r.url,
      completeness: r.completeness,
      updatedAt: r.updated_at,
    }));
  }

  close(): void {
    this.db.close();
  }
}
```

### Mapping to Convex Tables

| Convex Table | Storage Equivalent | Notes |
|---|---|---|
| `products` (name, url) | Metadata columns in `profiles` table | Flattened -- no separate products table |
| `productProfiles` (identity, revenue, entities, journey, definitions, outcomes, metrics, completeness) | `data` JSON column | Full document stored as JSON |
| `scanJobs` | Not in storage scope | Transient scan state lives in the CLI process |
| `crawledPages` | Not in storage scope | Consumed during analysis, not persisted in open source v1 |

The key simplification: Convex separates `products` and `productProfiles` because of its relational model (user -> product -> profile). The open source CLI has no user concept, so a profile is the top-level entity. Product metadata (name, url) becomes fields on the profile itself.

## Key Decisions

1. **JSON blob over normalized tables.** The `ProductProfile` schema has ~10 top-level sections, each with nested arrays and objects. Normalizing this into relational tables would require 15+ tables and a migration for every schema change. JSON blob with metadata columns gives us indexed search on the fields that matter (name, url) while making profile schema evolution free.

2. **Async interface, sync implementation.** `better-sqlite3` is synchronous, but the `StorageAdapter` interface uses `Promise` return types. This costs nothing (wrapping in `Promise.resolve` is near-zero overhead) and means Postgres/Turso adapters can be added without changing the interface contract.

3. **No ORM.** The schema is one table with five columns. Drizzle, Knex, or Prisma would add 50+ dependency packages for zero benefit. Raw SQL via `better-sqlite3`'s prepared statements is clearer and faster.

4. **No migration framework.** v1 has exactly one table. If the table shape changes, `CREATE TABLE IF NOT EXISTS` plus `ON CONFLICT` upsert handles it. The JSON blob means profile schema changes never require DDL changes. A migration framework can be added later if the SQLite schema grows.

5. **UUID over slug for IDs.** Slugs from product names would collide ("acme-inc" from two different Acme companies). UUIDs are unique without coordination. The name is available via the `name` column for human-readable listing.

6. **WAL mode by default.** `PRAGMA journal_mode = WAL` allows concurrent reads during writes. This matters for the MCP server which may handle multiple tool calls simultaneously.

7. **`close()` on the interface.** Resource cleanup is explicit. The CLI calls `close()` on exit. Tests call `close()` in teardown. This prevents SQLite lock file leaks.

## What This Does NOT Do

- **No Postgres adapter.** That is a community contribution target, not v1 scope.
- **No file-based adapter.** That is M008-E004-S002, a separate story.
- **No crawled page storage.** Crawled pages are consumed during analysis and discarded. Only the resulting profile is persisted.
- **No scan job tracking.** Scan progress is ephemeral CLI state, not persisted.
- **No user/auth model.** The open source CLI has no concept of users or authentication.
- **No LLM provider abstraction.** That is M008-E004-S003/S004, separate stories.
- **No full-text search.** `LIKE %query%` on name and url is sufficient. FTS5 can be added later if needed.
- **No schema migrations.** One table, JSON blob. Profile schema changes are free.

## Verification Steps

1. **Package builds:** `cd packages/storage && npx tsup` produces dist/ with .mjs, .cjs, .d.ts
2. **Type-check passes:** `cd packages/storage && npx tsc --noEmit` -- zero errors
3. **In-memory tests pass:** `cd packages/storage && npx vitest run` -- all tests use `:memory:` path
4. **save/load roundtrip:** Save a ProductProfile, load by ID, assert deep equality
5. **list returns metadata:** Save two profiles, call list(), verify summaries returned in updated_at order
6. **search by name:** Save profiles with different names, search by substring, verify correct results
7. **search by url:** Same as above but matching on URL
8. **delete works:** Save a profile, delete it, load returns null, delete again returns false
9. **auto-creates directory:** Construct with a path under a non-existent directory, verify it creates the directory and database file
10. **close is clean:** After close(), no lock files remain

## Success Criteria

- `StorageAdapter` interface exports: `save(profile)`, `load(id)`, `list()`, `delete(id)`, `search(query)`, `close()`
- `packages/storage/` has its own `package.json` with name `@basesignal/storage`
- `SQLiteStorage` creates database file at configurable path (default `~/.basesignal/data.db`)
- `save()` stores a `ProductProfile` and returns its ID
- `load()` retrieves a profile by ID, returns null if not found
- `list()` returns all profiles with basic metadata (id, name, url, updatedAt)
- `search()` finds profiles by product name or URL (basic text search)
- Tests use an in-memory SQLite database (no file I/O)
